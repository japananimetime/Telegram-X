## 3. Submodules — IMPORTANT (the tricky part)

The TDLib upgrade lives in submodule commits. The submodules only had upstream (read-only) remotes,
so forks were created and `.gitmodules` repointed:

| Submodule | Fork repo (PRIVATE) | Pinned commit / branch |
|-----------|---------------------|------------------------|
| `tdlib` | `japananimetime/tdlib` | `5c9512c` on `tgx/tdlib-flat` |
| `vkryl/td` | `japananimetime/tdlib-utils` | `4d92b85` on `tgx/td-richmessages-bindings` |

All other submodules (vkryl/core, vkryl/android, vkryl/leveldb, jni third_party, webrtc, tgcalls, …)
still point upstream and resolve fine (unchanged pins).

**Why `tdlib` is a FLAT (orphan, single-commit) snapshot:** the custom TDLib build's `.so` binaries
were Git-LFS-tracked. LFS push to a new repo **hung for ~14 h** (git-lfs's separate auth/handshake
stalls in this environment), and GitHub's server-side `GH008` rejected plain pushes because the full
unshallowed history referenced **664 LFS objects** not present on the fork. Fix: the 12 `.so`
(94 MB total, largest 25 MB — all under GitHub's 100 MB/file limit) were converted to **regular Git
blobs** (`.gitattributes` `*.so -text`, no `filter=lfs`) in a single orphan commit with no ancestry =
zero LFS references. A fresh clone of the pin gets real `.so` with **no LFS needed**.
- **DO NOT** re-enable LFS on the `tdlib` fork or try `git lfs push` — it hangs here.
- If the native build ever needs re-doing, rebuild `.so` and commit them as regular blobs again.
